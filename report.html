Теоретична частина

1. Теоретичні відомості

Electron — це відкрита кросплатформна платформа для розробки десктопних застосунків, створена компанією GitHub. Вона поєднує в собі рушій Chromium для відображення інтерфейсу та середовище виконання Node.js для доступу до системних ресурсів операційної системи. Завдяки цьому розробники можуть використовувати стандартні веб-технології — HTML, CSS та JavaScript — для створення повноцінних нативних застосунків під Windows, macOS і Linux.

1.1. Архітектура Electron. Electron-застосунок складається з двох ізольованих типів процесів.

  • Головний процес (Main Process) — єдиний процес, що виконується у середовищі Node.js. Він відповідає за створення вікон (BrowserWindow), управління життєвим циклом застосунку та взаємодію з файловою системою і операційною системою. Головний процес не має прямого доступу до DOM.
  • Процес рендерингу (Renderer Process) — кожне вікно BrowserWindow запускає власний процес рендерингу на основі Chromium. Цей процес відповідає за відображення HTML/CSS/JS-інтерфейсу та взаємодію з користувачем. За замовчуванням він не має доступу до API Node.js.

1.2. Безпека та Context Isolation. Сучасні вимоги безпеки Electron передбачають ізоляцію контексту між процесом рендерингу та Node.js-середовищем.

  • nodeIntegration: false — забороняє безпосередній доступ до Node.js API зі сторінки рендерингу, що захищає від XSS-атак.
  • contextIsolation: true — ізолює JavaScript-контекст preload-скрипту від контексту веб-сторінки.
  • preload.js — спеціальний скрипт, що виконується перед завантаженням сторінки у привілейованому контексті. Через contextBridge він дозволяє безпечно публікувати обмежений API для рендерера.

1.3. Міжпроцесна взаємодія (IPC). Для обміну даними між головним процесом і процесом рендерингу Electron надає модулі ipcMain та ipcRenderer.

  • ipcRenderer.invoke(channel, data) — надсилає асинхронний запит з процесу рендерингу до головного процесу і повертає Promise з відповіддю.
  • ipcMain.handle(channel, handler) — реєструє обробник у головному процесі, який отримує дані та повертає результат.

1.4. Робота з файловою системою. Модуль fs є частиною стандартної бібліотеки Node.js і надає функції для роботи з файлами. Метод fs.appendFileSync(path, data) дозволяє синхронно дописувати текст у кінець файлу, автоматично створюючи його у разі відсутності.

1.5. Структура проєкту. Типовий Electron-застосунок містить такі ключові файли: package.json (конфігурація проєкту та точка входу), main.js (головний процес), preload.js (міст між процесами), index.html (інтерфейс), renderer.js (логіка фронтенду) та styles.css (стилі).

main.js
const { app, BrowserWindow, ipcMain } = require('electron');
const path = require('path');
const fs = require('fs');

function createWindow() {
  const win = new BrowserWindow({
    width: 600,
    height: 500,
    webPreferences: {
      nodeIntegration: false,
      contextIsolation: true,
      preload: path.join(__dirname, 'preload.js')
    }
  });

  win.loadFile('index.html');
}

ipcMain.handle('submit-survey', async (_event, data) => {
  const { name, age, language } = data;
  const timestamp = new Date().toLocaleString();
  const entry = `--- Survey Entry (${timestamp}) ---\n` +
                `Name: ${name}\nAge: ${age}\n` +
                `Favorite Language: ${language}\n\n`;
  const filePath = path.join(__dirname, 'survey_answers.txt');

  fs.appendFileSync(filePath, entry, 'utf-8');
  return { success: true };
});

app.whenReady().then(createWindow);

app.on('window-all-closed', () => {
  if (process.platform !== 'darwin') app.quit();
});

app.on('activate', () => {
  if (BrowserWindow.getAllWindows().length === 0) createWindow();
});
preload.js
const { contextBridge, ipcRenderer } = require('electron');

contextBridge.exposeInMainWorld('surveyAPI', {
  submitSurvey: (data) => ipcRenderer.invoke('submit-survey', data)
});

renderer.js
const form = document.getElementById('surveyForm');
const messageDiv = document.getElementById('message');

form.addEventListener('submit', async (e) => {
  e.preventDefault();

  const data = {
    name:     document.getElementById('name').value.trim(),
    age:      document.getElementById('age').value.trim(),
    language: document.getElementById('language').value.trim()
  };

  try {
    const result = await window.surveyAPI.submitSurvey(data);
    if (result.success) {
      showMessage('Answers saved successfully!', 'success');
      form.reset();
    }
  } catch (err) {
    showMessage('Error saving answers. Please try again.', 'error');
  }
});

function showMessage(text, type) {
  messageDiv.textContent = text;
  messageDiv.className = `message ${type}`;
  setTimeout(() => {
    messageDiv.className = 'message hidden';
  }, 3000);
}

Результат

Після запуску застосунку командою npm start відкривається вікно форми опитування.

Форма складається з трьох полів: Name (текстове поле для введення імені), Age (числове поле з обмеженням від 1 до 120) та Favorite Programming Language (текстове поле з автодоповненням через <datalist> — під час введення тексту відображається список з 20 популярних мов програмування, проте користувач може ввести будь-яке значення вручну).

Після заповнення всіх полів та натискання кнопки Submit дані через IPC-канал передаються у головний процес. Головний процес формує рядок запису з міткою часу та дописує його у файл survey_answers.txt в кореневій теці проєкту. Інтерфейс відображає зелене повідомлення про успішне збереження, після чого форма очищується.

Приклад вмісту файлу survey_answers.txt після відправлення форми:

--- Survey Entry (17.04.2026, 19:30:00) ---
Name: Roman
Age: 21
Favorite Language: JavaScript

Висновок

Під час виконання другої лабораторної роботи я успішно опанував основні концепції розробки десктопних застосунків на платформі Electron. Практичним результатом роботи стало створення застосунку «Форма опитування», що дозволяє збирати відповіді користувача на три запитання та зберігати їх у локальний текстовий файл.

У процесі розробки я навчився правильно організовувати структуру Electron-проєкту з розподілом відповідальності між головним процесом та процесом рендерингу. Особливу увагу було приділено безпечній архітектурі: відключено nodeIntegration, увімкнено contextIsolation, а весь обмін даними між процесами реалізовано виключно через IPC-механізм (ipcMain.handle / ipcRenderer.invoke) з використанням preload-скрипту та contextBridge. Для покращення зручності використання поле вибору мови програмування реалізоване у вигляді текстового поля з підказками автодоповнення (<datalist>), що дозволяє як обрати значення зі списку, так і ввести власне.

Порівняно з підходом WinForms/WPF з першої лабораторної роботи, розробка на Electron демонструє значні переваги у гнучкості оформлення інтерфейсу завдяки використанню CSS, а також кросплатформність застосунку без зміни коду.
